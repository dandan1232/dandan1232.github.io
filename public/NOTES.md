# 会话笔记

## 2026-03-05

### 数字人重复播放问题复盘

#### 问题现象
- 同一段播报内容被触发多次，出现连续重复播放。
- 在网络抖动或快速连续触发时，重复概率明显上升。

#### 根因分析
- 播放状态未串行化：新请求到来时，旧播放流程还未完全结束。
- 事件回调重复绑定：`ended` / 消息监听器被多次注册，导致一次完成触发多次后续逻辑。
- 消息缺少幂等控制：同一条播报任务在重试或重连后被再次消费。

#### 解决方案
- 引入播放会话标识（`playSessionId`）：
  - 每次开始新播放生成新会话号。
  - 回调执行前先校验会话号，不匹配直接丢弃。
- 增加播放互斥锁（`isPlaying` + `isSwitching`）：
  - 确保同一时刻只允许一个有效播放流程。
  - 切换时先停旧流程，再启动新流程。
- 做监听器去重与清理：
  - 监听器统一在初始化时绑定一次。
  - 切换/销毁时显式移除旧监听，避免重复触发。
- 增加消息幂等去重：
  - 基于 `messageId`（或内容哈希）过滤已消费任务，防止重复入队。

#### 验证结果
- 连续触发、重连恢复、弱网重试场景下，重复播放问题消失。
- 播放链路行为稳定：只保留“最新会话”有效输出。

#### 经验沉淀
- 音视频播放链路必须同时具备：
  - 会话隔离（token/session）
  - 流程互斥（lock）
  - 事件生命周期管理（bind once + cleanup）
  - 消息幂等（dedupe）
