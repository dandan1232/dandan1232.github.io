从 0 到 1 构建 RLHF 系统

整理于 2026-05-07 · 来源：小红书大模型团队 QCon 分享

本文整理自小红书 AGI 团队在 QCon 上海 2024 的分享，介绍其自研 MLLM RLHF 训练框架的设计与优化，覆盖异构组网、训推一体、PPO 细节处理和 Medusa 加速等核心技术。

RLHF 背景

RLHF（基于人类反馈的强化学习）是让大模型输出更贴近人类偏好的关键技术。核心流程：

SFT（前置阶段） ↓ 第一阶段：RM（Reward Model） → 偏好训练一个打分模型，模拟人类视角评价数据质量 ↓ 第二阶段：RL（PPO） → LLM 根据指令生成回复 → RM 实时打分 → 策略优化 → 不断生成接近人类偏好的输出

PPO 核心组件

  • Actor Model — 演员模型，训练的目标语言模型
  • Critic Model — 评论家模型，预估总收益
  • Reward Model — 奖励模型，计算即时收益
  • Reference Model — 参考模型，防止模型训歪
PPO 是一个训推混合流程：第一阶段 Make-Experience（Rollout）涉及 4 个模型推理；第二阶段 Training 涉及 actor 和 critic 两个模型独立训练。工程痛点：多阶段 dataloader、generate 自回归、4 模型推理 + 2 模型训练。

框架设计

技术选型

组件选型原因
训练框架Megatron-core + Ray降低维护成本，避免缝合怪
推理引擎vLLM高性能推理，但 logp 需走 Megatron 保证一致性

异构组网架构

核心思路：Forward-Offload 分时复用模型，将 4 个模型的显存降到 2 个。

原始方案：4 模型 × 8机64卡 = 32机256卡 ↓ Forward-Offload Offload 方案：actor→offload→ref, critic→offload→RM → 2 模型 × 8机64卡 = 16机128卡（降低 50%） ↓ Training-Separated actor / critic 独立集群异步训练 → 吞吐最大化
启动方式：复用 torchrun 拉起主任务，实现 mock-run 方式自动拉起远端 critic 集群。

同构组网架构

当数据量和模型规模增大时，进一步压缩到单个 SFT 所需的训练规模：

  • Forward 阶段：offload param 完成模型切换
  • Training 阶段：offload 中间状态（param/grad/ddp-bucket/opt-state）
  • 代价：增加上下文切换开销，但相对训练时间较小

训推一体优化

Pipeline 优化

问题：actor 采样阶段训练空窗，资源浪费。

方案特点
常规版本推理/训练独立集群，存在优化空间
全量 OffloadvLLM/actor/critic 全集群资源利用，ROI 高，但有 offload/restart 空隙
Pipeline 版本流水线 overlap 数据落盘、restart/offload、部分 forward 开销，性能最优

PPO 细节处理

  • Padding-Free 优化：对 generate/forward/train 三阶段去除 padding，整体提升 10%
  • 参数同步：提前与 Ray-vLLM 组建 NCCL Group，比文件系统 reload 快几十秒
  • CP 并行优化：规避对 logits 进行 gather，先统一得到 total logp，通信量从 BSH 降到 BS

训推一致性

RM 的 vhead 本质是矩阵乘 [1,hidden] × [hidden,1]，初始 diff ~0.001 经过矩阵乘后放大到 0.3，精度无法容忍。解决方案：RM+RL 使用相同训练框架；RL 场景下 RM-serving 转为本地 offload forward。

Medusa 投机采样

通过多个 head 预测降低 decode 次数，无损优化自回归性能。核心设计：

  • 主干网络 + 多个 Medusa Head（与 LM-Head 同结构）
  • RL 场景关注吞吐，running-batch 尽可能大
  • 伴生训练：backbone 实时更新，mhead 延后更新，可与训练阶段 overlap
  • 即使 running-batch 打满，仍有 50% 性能收益

多模态优化

  • 组网优化：LLM 的 TP/PP/CP 用作视觉部分的 DP
  • 多路复用：vLLM/actor/critic 复用相同的 img_feature
  • Prefetch：frozen visual-model 通过 prefetch 融入 make-experience 做 Pipeline overlap

实践效果

  • 开源模型 RL 后相比基线 SFT GSB 提升 5-20%
  • 推理、数学提升较多，综合自动化评分 提升 6%
  • PRM（过程奖励）让激励更准确，reward-gap 加大，提升模型上限 5%

调参经验

  • 可视化逐样本/token 进行细粒度分析
  • Advantage-batch 对小 DP 场景有用，防止走偏
  • Critic 从 RM 参数加载，复用训练集群做 reward 的 batch-forward
  • LR 基于 SFT 设置，actor < critic，否则容易发散
  • Critic 先学习，actor 先 frozen，更准确地学习
  • Reward Hacking：训练指标持续增加但评测下降时需要迭代 RM
