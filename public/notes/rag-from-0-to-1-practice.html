RAG 从 0 到 1 落地笔记

发布于 2026-03-05

目标定义

  • 回答要准：降低幻觉和错引。
  • 过程可解释：给出来源片段与引用路径。
  • 系统可迭代：支持持续加数据、调策略、看指标。

核心架构

  • 数据层：结构化库 + 文档库 + 图谱库（实体/关系）。
  • 检索层：关键词检索、向量检索、图关系检索并行召回。
  • 排序层：轻量重排模型对候选片段打分，保留高相关上下文。
  • 生成层：LLM 严格基于检索上下文回答，并输出引用证据。

推荐流程

  • 问题理解：先做意图分类，区分事实问答、流程问答、总结问答。
  • 多路召回：BM25 + Vector + Graph 同时召回，避免单路偏差。
  • 重排截断：按相关性排序后截断 token，保留最有信息密度的片段。
  • 答案约束：提示词中强制“无依据就说不知道”，减少编造。
  • 可追溯输出：返回答案 + 引用片段 + 文档ID/实体ID。

评估与监控

  • 离线指标：召回率、MRR、nDCG、答案正确率。
  • 在线指标：首答时延、用户追问率、人工纠错率。
  • 样本池建设：保留高频问题、失败样本和边界问题，持续回归测试。

常见坑位

  • 只做向量检索：术语精确匹配场景会丢召回。
  • 分块过大或过小：过大噪声多，过小语义断裂。
  • 无版本化：知识更新后无法追踪旧答案来源。
  • 缺少拒答策略：模型会在低置信度场景硬答。

落地结论

RAG 不是“接个向量库就结束”。真正可用的系统必须把数据治理、检索策略、答案约束和评估闭环一起建设，才能在业务场景里稳定输出。
