Easy-Langent：从 LangChain 到 LangGraph 的智能体开发之路

发布于 2026-05-07 · Agent 开发

前言

"easy-langent" 是一个系统性的 AI 智能体开发教程，由 "lang"（LangChain、LangGraph 等框架）与 "agent"（智能体）合并命名。核心目标是打破"理论学习"与"实战开发"的壁垒，让读者从懂概念到会开发。

  • 项目地址：github.com/datawhalechina/easy-langent
  • 前置要求：Python 基础、大模型基本概念、智能体核心概念
  • 建议周期：5 周系统学习

一、为什么需要 LangChain 和 LangGraph

假设你要做一个"智能论文助手"，没有框架时需要自己处理：不同厂商 API 的差异、对话历史管理、工具调用、多步骤流程管控……这些是传统开发的三大痛点：

  • 重复造轮子：每次开发都要重写模型调用、对话管理等通用代码
  • 流程管控复杂：多步骤任务容易出错
  • 状态维护困难：中间数据不好保存和传递

LangChain 和 LangGraph 就是为了解决这些麻烦而生的工具。

二、两大框架的定位与关系

2.1 LangChain：大模型应用的"乐高积木"

LangChain 的定位是帮你快速搭建简单到中等复杂度的大模型应用。它把通用功能封装成组件（模型调用、对话管理、工具连接等），你只需拼装即可。

适用场景：简单 LLM 调用、基础 RAG、单步骤工具调用、快速验证想法。

三层架构（1.0+）

模块职责
langchain_core核心抽象（Runnable、BaseParser 等）
langchain高级工程组件（Chains / Parsers / Memory）
langchain_openai第三方模型适配

这种分层设计的优势：核心团队专注稳定性，社区可开放共建集成包，开发者灵活选择安装。

2.2 LangGraph：复杂应用的"建筑设计图"

LangGraph 基于 LangChain 生态，专门处理多步骤、需要协作的复杂任务。它以状态机为核心，支持流程管控、状态管理和多智能体协作。

适用场景：多步骤流程、需要保存中间结果、多智能体协作、需要人机交互的流程。

2.3 核心关系

  • 从属关系：LangGraph 是 LangChain 生态的进阶扩展，依赖 LangChain 核心组件
  • 互补关系：简单任务用 LangChain（快、简单），复杂任务用 LangGraph（稳、可控）

三、LangChain 核心组件

3.1 模型调用（ChatOpenAI）

LangChain 提供统一接口调用不同厂商的模型。两种模型类型：LLM（文本生成）和 ChatModel（对话）。通过角色消息（system / user / assistant）构建对话，角色代表约束层级而非简单的说话人。

from langchain_openai import ChatOpenAI

llm = ChatOpenAI(
    api_key=API_KEY,
    base_url=BASE_URL,
    model="deepseek-chat",
    temperature=0.3
)
response = llm.invoke("请写一段 AI 学习建议")

3.2 提示词模板

  • PromptTemplate：将固定文本与动态参数分离，用 {variable} 占位
  • FewShotPromptTemplate：提供示例引导模型输出格式，适合结构化文档生成
  • ExampleSelector：动态选择示例，支持基于长度的选择策略

3.3 输出解析

将 LLM 的非结构化输出转换为程序可用的结构化数据：

解析器用途
StrOutputParser将 AIMessage 转为纯字符串，所有系统的基础
JsonOutputParser快速演示场景，输出键值对字典
PydanticOutputParser生产系统，通过 Pydantic 模型做强类型验证

四、LangGraph 核心概念

4.1 State（状态）—— 共享黑板

状态是工作流的"共享数据容器"，用于传递和存储中间结果。LangGraph 1.0+ 推荐用 TypedDict 定义状态结构，生产环境推荐 Pydantic 的 BaseModel。

from typing import TypedDict

class WorkflowState(TypedDict, total=False):
    user_role: str          # 用户角色
    original_advice: str    # 原始建议
    simplified_advice: str  # 精简建议

状态传递遵循"不可变更新"原则：节点接收状态副本，执行逻辑，返回更新的字段，框架自动合并为新的全局状态。

4.2 Node（节点）—— 功能工人

节点是执行单元，本质是纯函数："输入状态 → 处理 → 返回更新状态"。三种类型：

  • LLM 调用节点：通过大模型生成内容
  • 工具调用节点：对接外部工具（搜索、API）
  • 数据处理节点：转换、验证、格式化数据
def generate_advice(state: WorkflowState):
    prompt = f"给{state['user_role']}写一段50字的 AI 学习建议。"
    result = llm.invoke(prompt)
    return {"original_advice": result.content}

4.3 Edge（边）—— 路径导航

三种边类型：

  • 固定边（add_edge）：无条件的线性跳转
  • 条件边（add_conditional_edges）：基于状态的动态路由
  • 循环边：条件边的特殊形式，路由回前序节点直到终止条件

4.4 Hello World：两步工作流

from langgraph.graph import StateGraph, START, END

# 构建工作流
workflow = StateGraph(WorkflowState)
workflow.add_node("generate", generate_advice)
workflow.add_node("simplify", simplify_advice)

workflow.add_edge(START, "generate")
workflow.add_edge("generate", "simplify")
workflow.add_edge("simplify", END)

app = workflow.compile()

# 执行
result = app.invoke({"user_role": "高校学生"})
print("原始建议：", result["original_advice"])
print("精简建议：", result["simplified_advice"])

五、LangGraph 进阶：状态工作流

5.1 超步执行机制

LangGraph 基于"超步"（Super-Step）执行：每一轮并行激活接收消息的节点，执行所有活跃节点，将新状态片段传递给下游节点。支持顺序执行和平行执行。

5.2 分支工作流：质量验证循环

在文本处理管道中加入质量检查：如果摘要不通过长度/内容验证，循环回去重新生成（最多重试 2 次），然后强制输出。关键设计：

  • 状态中增加 rewrite_count 和 quality_valid 字段
  • 路由函数根据质量检查结果选择 to_output、to_rewrite 或 to_force_output

5.3 人机协作：交互式文本优化

一个交互式编辑助手：用户输入文本 → AI 优化 → 展示结果和建议 → 用户选择"确认"（结束）、"修改"（循环）或"退出"。关键特性：

  • feedback 节点使用 input() 暂停工作流等待用户输入
  • 优化节点根据是否有用户反馈自适应调整 Prompt
  • 使用 MemorySaver 做状态持久化

六、学习路径建议

周次核心任务对应内容
第 1 周框架认知与环境搭建前言 + 第一章
第 2 周LangChain 核心能力第二到五章（模型调用、提示词、输出解析、记忆与工具）
第 3 周LangGraph 基础与进阶第六到七章（状态工作流、多智能体）
第 4 周综合实战第八章（完整项目开发）

七、总结

easy-langent 课程的核心价值不仅在于掌握两个框架，更在于传递一种"模块化、工程化、场景化"的 AI 应用设计思维：

  • LangChain 是"基础设施工具箱"——灵活、可复用的核心组件，降低开发门槛
  • LangGraph 是"架构设计框架"——解决复杂任务的流程管控、状态管理与多角色协作
  • "状态机设计""工具调用逻辑""多智能体协作模式"这些底层核心不会随框架版本过时

掌握这些思想，即便未来框架更新，只需快速浏览更新日志就能适配——就像学 Python，语法有小更新，但变量、循环、函数的核心概念从未改变。
